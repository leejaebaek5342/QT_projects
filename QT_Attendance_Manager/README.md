# Qt로 출석 관리 프로그램 만들기

## Overview
Qt Widgets 기반 출석 관리 프로그램으로,  
교사 계정별 학생 및 출석 데이터를 분리 관리하고  
출석 상태 저장 및 조회 기능을 구현하였다.

## 이 프로그램은 다음과 같은 흐름으로 동작한다

- 메인 화면에서 Teacher / Student 모드 선택
- Teacher 회원가입 및 로그인 진행
- 로그인한 교사의 ID를 기준으로 데이터 파일 생성 및 로드
- 학생 등록 및 목록 관리
- 날짜별 출석 상태 저장
- 저장된 출석 데이터를 기반으로 학생별 출석 조회
- Student 모드에서는 이름과 학번을 입력하여 해당 학생의 출석 기록 조회

## 프로그램 구성

프로그램은 `Teacher 영역`과 `Student 영역`으로 구분하여 설계하였다.

- Teacher 영역  
  회원가입, 로그인, 학생 등록, 학생 목록 확인, 출석 저장, 학생별 출석 조회 기능을 담당한다.

- Student 영역  
  이름과 학번을 입력하여 본인의 출석 기록을 조회할 수 있도록 구성하였다.

Teacher 영역에서 등록한 학생 및 출석 데이터는 로그인한 교사의 ID를 기준으로 분리 저장되며,  
Student 영역에서는 입력한 이름과 학번이 일치하는 경우에만 해당 출석 기록을 조회할 수 있도록 하였다.

## 구현 방식

### 화면 구성

- `QStackedWidget`을 이용해 Teacher 화면과 Student 화면 전환을 처리하였다.
- Teacher 회원가입은 별도 위젯으로 분리하여 새로운 창으로 실행되도록 구현하였다.

### 주요 UI 구성

- 학생 목록은 `QTableWidget`에 출력하였다.
- 출석 상태는 각 학생 행마다 `QComboBox`를 배치하여 선택할 수 있도록 구성하였다.

### 데이터 저장

- 교사 계정 정보는 `teachers.txt`에 저장하였다.
- 학생 목록과 출석 정보는 로그인한 교사의 ID를 기준으로 파일을 분리하였다.
- 학생 목록: `studentslist_<teacherId>.txt`
- 출석 정보: `attendance_<teacherId>.txt`

### 데이터 처리

- 로그인한 교사의 ID를 기준으로 학생 목록과 출석 파일을 로드하였다.
- 출석 저장 시 선택된 날짜와 학생별 출석 상태를 파일에 기록하였다.
- Student 모드에서는 이름과 학번이 일치하는 학생의 출석 기록만 조회되도록 처리하였다.

## 문제 해결

### 1. 교사별 데이터 분리 문제

- 문제  
  학생 목록과 출석 정보를 공용 파일 하나로 관리하여,  
  교사가 바뀌어도 이전 교사의 학생 목록이 그대로 보이는 문제가 발생하였다.

- 원인  
  학생 목록과 출석 데이터가 로그인한 교사 ID와 연결되지 않고  
  동일한 파일에 저장되고 있었기 때문이었다.

- 해결  
  로그인한 교사의 ID를 기준으로 파일명을 분리하여  
  `studentslist_<teacherId>.txt`, `attendance_<teacherId>.txt` 형식으로 저장하도록 수정하였다.

---

### 2. 함수 반환값 처리 문제

- 문제  
  중복 검사나 학생 정보 확인 함수에서 `bool`을 사용하면  
  “데이터 없음”과 “파일 열기 실패”를 구분하기 어려웠다.

- 해결  
  `int` 반환값을 사용하여 상태를 세분화하였다.

  - `1`: 중복 있음
  - `0`: 중복 없음
  - `-1`: 파일 열기 실패 또는 교사 정보 없음

- 결과  
  실패 원인에 따라 다른 메시지를 출력하고,  
  예외 상황을 더 명확하게 분기 처리할 수 있었다.
  
## 느낀점

이번 미니 프로젝트를 만들면서 Qt에서 여러 위젯을 연결해 프로그램 흐름을 구성하는 방법을 연습할 수 있었다.  
특히 단순히 화면만 구현하는 것이 아니라, 로그인 정보와 데이터 저장 구조까지 함께 생각해야 한다는 점이 어렵기도 했다.

이번에는 텍스트 파일 기반으로 구현했지만,  
학생 수나 데이터가 더 많아진다면 파일을 직접 관리하는 방식보다는 DB를 활용하는 것이 더 효율적일 것 같다는 생각이 들었다.

또 이번 프로젝트에서는 대부분의 기능을 하나의 `widget.cpp` 안에서 처리했는데,  
기능이 많아질수록 코드가 길어지고 읽기 어려워진다는 점도 느꼈다.  
실습평가 때는 다양한 기능을 하나의 클래스에 몰아넣기보다, 역할별 클래스로 분리해 구현하는 것을 목표로 하고 있다.
